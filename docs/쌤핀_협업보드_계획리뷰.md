# 쌤핀 협업보드 구현계획 — 종합 리뷰 보고서

> **리뷰 일자**: 2026-04-19
> **리뷰 방식**: Ultrawork 5개 병렬 에이전트 (Opus×3 + Sonnet×2)
> **리뷰 대상**: 쌤핀_협업보드_구현계획.md

---

## 총평: B+ (양호하나 구현 전 보완 필수)

기술 선택(Excalidraw + Y.js)은 적절하고, 아키텍처 방향(교사 PC = 서버)도 쌤핀에 맞다. 그러나 **쌤핀 고유 아키텍처 관습과의 정합 설계**, **보안/데이터 리스크**, **Phase 1 범위 과다**라는 세 가지 핵심 문제가 구현 전에 반드시 해결되어야 한다.

---

## 1. 기술적 실현 가능성

| 항목 | 판정 | 핵심 이슈 |
|------|------|-----------|
| Excalidraw v0.18 + Y.js | ⚠️ 주의 | y-excalidraw가 v0.18 호환 미검증, npm 미발행 |
| y-websocket Electron import | ⚠️ 주의 | ESM/CJS 충돌 → ws 직접 사용으로 우회 가능 |
| cloudflared WebSocket | ⚠️ 주의 | 20~30초 유휴 시 연결 끊김 다수 보고 |
| 30명 Y.js CRDT 성능 | ✅ 가능 | Y.js 벤치마크상 충분 |
| Excalidraw 번들(~2.3MB) | ✅ 가능 | Electron은 로컬 로드, 학생 앱만 캐싱 필요 |
| 페이지 시스템 API | ✅ 가능 | 단, 페이지별 Y.Doc 분리 설계 필요 |
| CSS 해킹 (커스텀 툴바) | ⚠️ 주의 | 내부 클래스명은 semver 보장 안 됨 |

### 필수 조치 3건

**① y-excalidraw 의존 제거** → 자체 바인딩 구현 권장
- y-excalidraw는 1인 개발, npm 미발행, v0.18 호환 미검증
- Y.Array ↔ Excalidraw elements 직접 바인딩은 복잡도 낮음
- 참고: excalidraw-yjs-starter 패턴

**② cloudflared heartbeat 필수** → 25초 간격 ping/pong
- 유휴 시 Cloudflare가 WebSocket 연결 끊는 사례 다수
- Y.js 동기화 메시지가 자연 heartbeat이지만 유휴 시 보장 안 됨

**③ 페이지별 Y.Doc 분리** → resetScene()이 Y.Doc에 "전체 삭제" 전파 방지
- 페이지 전환 시 다른 사용자의 캔버스까지 초기화되는 버그 가능

## 2. 아키텍처 적합성

### 미준수 항목 6건

**① Clean Architecture 레이어 미반영** (심각)
- 계획서 코드가 레이어 구분 없이 평면적
- 필요: `Board` 엔티티, `IBoardRepository` 포트, `ManageBoard` 유스케이스, `useBoardStore` 어댑터

**② electron/ipc/ 패턴 불일치**
- 기존 5개 라이브 도구: `registerXxxHandlers(mainWindow)` + 싱글턴 session + ipcMain.handle
- 계획서: 독립 모듈 형태 → `registerBoardHandlers(mainWindow)` 패턴으로 통일 필요

**③ Zustand 스토어 설계 누락**
- `useBoardStore` (보드 목록 CRUD) + `useBoardSessionStore` (실시간 세션 상태) 필요

**④ preload.ts contextBridge 미고려**
- 보드 추가 시 15~20개 IPC 메서드 추가 필요 → 네임스페이스 설계 필요

**⑤ 학생 웹앱 빌드 파이프라인 누락** (심각)
- 기존: 인라인 HTML 생성 (liveSurveyHTML.ts 등)
- 보드: Excalidraw ~2MB → 인라인 불가, Vite 별도 빌드 필수
- electron-builder의 `extraResources`로 패키징 설정 필요

**⑥ 터널 싱글턴 충돌** (심각)
- `tunnel.ts`의 `activeTunnel`이 전역 싱글턴
- 보드 + 투표/설문 동시 사용 시 터널이 덮어씌워짐
- 멀티 터널 지원 또는 상호 배타 정책 필요

## 3. 누락된 리스크

### 높음 (3건)

**① 보안: 인증 없는 WebSocket 서버**
- 터널 URL만 알면 누구나 접속하여 Y.Doc 덮어쓰기 가능
- 완화: URL에 무작위 토큰 포함 + 연결 시 1회용 코드 검증 + 최대 연결 수 제한

**② 네트워크: WiFi AP 병목**
- 교실 AP 1대로 30명 WebSocket 지속 연결 시 동시 세션 한도 초과 가능
- 완화: 재연결 지수 백오프 + LAN 직접 접속 1차 폴백

**③ 데이터 유실: 자동 저장 미구현**
- 코드 스니펫의 `saveBoardToFile`이 주석 처리 상태
- PC 크래시 시 메모리의 Y.Doc 소멸
- 완화: 30초 간격 자동 저장 + `before-quit` 동기 저장

### 중간 (3건)

**④ 의존성**: y-excalidraw 소규모 프로젝트 + `"latest"` 버전 고정 없음
**⑤ 교육 현장**: 터널 끊김 시 30명 동시 재연결 + 크롬북 저사양 성능
**⑥ 법적(PIPA)**: WebSocket 연결 시 학생 IP/접속 로그 수집 문제

## 4. UX 완성도

| 항목 | 판정 | 핵심 이슈 |
|------|------|-----------|
| 학생 온보딩 | 🔧 개선 | 입장 화면 상세 스펙 없음 (중복 이름, 피드백) |
| 교사 워크플로우 | 👍 양호 | 종료 후 저장 확인 UI만 추가 |
| Jamboard 전환 UX | 🔧 개선 | 페이지 전환 시 학생 화면 동기화 불명확, "홈 버튼" 필요 |
| 모바일/태블릿 | ❓ 누락 | **팜 리젝션 미고려** — iPad + Pencil 필수 시나리오 |
| 접근성 | ❓ 누락 | 색각 이상(남학생 5~6%), 스크린 리더, 최소 폰트 크기 |
| 에러 처리 UI | 🔧 개선 | 학생용 연결 끊김 안내 화면 없음 |
| 30개 보드 모니터링 | 🔧 개선 | 썸네일 크기, 확대 클릭, 완료 표시 UX 미정의 |

### 누락된 Jamboard 핵심 기능

- **실시간 커서 공유**: Y.js awareness API로 구현 가능하나 계획에 없음 → 협업 실재감에 직결
- **이미지 검색 삽입**: Jamboard 핵심 기능, Excalidraw는 로컬 업로드만 지원
- **손글씨 인식**: 초등 수업 활용도 높음, Phase 4 이후 검토

## 5. 로드맵 현실성

### Phase 1 범위: 비현실적

현재 Phase 1에 11개 작업이 있으며, Jamboard 커스터마이징 3건이 추가되어 14개. 사이드 프로젝트 수준(주 15~20시간)에서 2~3주는 불가능하다.

### 개선된 로드맵 제안

```
Phase 1a (3주) — 최소 동작 보드
├─ Excalidraw 기본 UI (roughness:0만 적용)
├─ Y.js WebSocket 서버 (ws 직접 구현)
├─ cloudflared 터널 + QR
├─ 학생 웹앱 Vite 빌드 파이프라인
├─ 자동 저장 / 불러오기
├─ Clean Architecture 레이어 설계
├─ preload.ts IPC 채널 추가
└─ 검증: "30명이 같이 그릴 수 있다"

Phase 1b (2주) — Jamboard 스타일링
├─ 커스텀 좌측 툴바
├─ 스티키 노트
├─ 페이지 시스템 (페이지별 Y.Doc 분리)
├─ 색상 팔레트 단순화
├─ 모바일 터치 기본 대응 (팜 리젝션)
└─ 학생 입장 화면 + 에러 처리 UI

Phase 2 (2~3주) — 교사 제어
├─ 역할 구분 + 보드 잠금
├─ 도구 제한 + 포커스 모드
├─ Y.js awareness (실시간 커서 공유)
├─ 타이머 + 내보내기
└─ 보안: 토큰 인증 + 연결 수 제한

Phase 3 (2~3주) — 개인/그룹 보드
├─ 멀티 room + 교사 대시보드
├─ 그룹 편성 + 학생명단 연동
├─ 콘텐츠 배포 + 스냅샷
└─ 접근성 기본 대응

Phase 4 (2주) — 고급 기능
├─ 보드 템플릿
├─ 히스토리 재생
├─ Google Drive 백업
└─ 이미지 검색 삽입 (선택)
```

### 핵심 변경점
- Phase 1을 1a/1b로 분할 → 1a 완료 시 즉시 교실 테스트 가능
- 모바일 터치를 Phase 4 → Phase 1b로 격상 (학생이 태블릿으로 접속하는 게 전제)
- Y.js awareness(커서 공유)를 Phase 2에 명시
- 보안(토큰 인증)을 Phase 2에 명시
- 터널 싱글턴 충돌 해결을 Phase 1a에 포함 필요

## 7. 결론

계획서의 기술 방향은 올바르다. Excalidraw + Y.js + 교사 PC 서버 모델은 서버리스 제약을 잘 해결한다. 그러나 **쌤핀 아키텍처 관습(Clean Architecture, IPC 패턴, 터널 싱글턴)과의 정합 설계가 빠져 있고**, **보안/데이터 리스크가 과소평가**되어 있으며, **Phase 1 범위가 과다**하다.

위 체크리스트의 Blocker 5건을 해결한 후 Phase 1a부터 착수하면, 3주 내에 "30명이 같이 그리는" MVP를 교실에서 테스트할 수 있다.
