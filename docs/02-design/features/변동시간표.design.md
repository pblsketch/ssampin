## 1. Architecture Overview

### 1.1 Design Goals

- 숨어있던 `TimetableOverride`를 **명시적 UI + 관리 패널**로 승격해 발견성 확보
- `addOverride` upsert·`updateOverride`·`getEffectiveClassSchedule`를 통해 **데이터 무결성과 적용 범위 일관성** 확보
- 병합 로직은 `domain/rules` 순수 함수로 분리 → **스토어는 호출만, 테스트는 순수 함수만**
- NEIS 자동 재동기화가 override를 **절대 건드리지 않음**을 코드·테스트로 보장
- Clean Architecture 4-layer 의존성 방향 엄수: `domain ← usecases ← adapters ← infrastructure`

### 1.2 Design Principles

- **순수 함수 우선**: 병합·필터·upsert 판정은 `domain/rules/timetableRules.ts` 확장에서 처리
- **기존 구조 최소 변경**: `TempChangeModal`은 props 확장으로 재사용, `JsonScheduleRepository`는 그대로
- **Backward compatible**: `TimetableOverride`는 `updatedAt` optional 필드만 추가 (기존 JSON 파일 호환)
- **Silent failure 금지**: upsert 시 교체된 경우 Toast로 "기존 변동을 교체했습니다" 안내 (선택)
- **Cross-platform**: Electron + 브라우저 모두에서 동일 동작 (storage 추상화 재사용)

**파일**: [src/domain/entities/Timetable.ts](../../../src/domain/entities/Timetable.ts)

```ts
export interface TimetableOverride {
  readonly id: string;
  readonly date: string;           // "YYYY-MM-DD"
  readonly period: number;         // 1-based
  readonly subject: string;        // "" = 공강/자습
  readonly classroom?: string;
  readonly reason?: string;
  readonly createdAt: string;
```

## 3. Domain Pure Functions (신규)

**파일**: [src/domain/rules/timetableRules.ts](../../../src/domain/rules/timetableRules.ts)에 추가

### 3.1 `upsertOverride`

```ts
/**
 * overrides 배열에 새 항목을 upsert한다.
 * 같은 date+period가 이미 존재하면 id는 기존 id를 유지하고 값만 교체 (updatedAt 갱신).
 * 없으면 새로 추가.
 *
 * @param existing  현재 overrides 배열 (불변)
 * @param input     date, period 포함 전체 필드 (id, createdAt, updatedAt 제외 가능)
 * @param now       현재 시각 ISO 문자열 (테스트 주입용)
 * @param idFactory id 생성기 (테스트 주입용)
 * @returns         { overrides: 새 배열, replacedId: 교체된 경우 해당 id | null }
 */
export function upsertOverride(
  existing: readonly TimetableOverride[],
  input: Omit<TimetableOverride, 'id' | 'createdAt' | 'updatedAt'>,
  now: string,
  idFactory: () => string,
): { overrides: readonly TimetableOverride[]; replacedId: string | null };
```

### 3.2 `mergeOverridesIntoTeacherSchedule` / `mergeOverridesIntoClassSchedule`

```ts
/** 교사용 유효 시간표 계산 (기존 getEffectiveTeacherSchedule의 로직을 순수 함수로 추출) */
export function mergeOverridesIntoTeacherSchedule(
  base: readonly (TeacherPeriod | null)[],
  overridesForDate: readonly TimetableOverride[],
): readonly (TeacherPeriod | null)[];

/** 학급용 유효 시간표 계산 (신규) */
export function mergeOverridesIntoClassSchedule(
  base: readonly ClassPeriod[],
  overridesForDate: readonly TimetableOverride[],
): readonly ClassPeriod[];
```

**규칙**:
- `period` 인덱스가 범위 밖이면 무시
- `override.subject === ""` → 교사용은 `null`, 학급용은 `{ subject: '', teacher: '' }` (공강)
- 교사 포함: `override.subject` + `override.classroom ?? ''`
- 학급 포함: `override.subject` + `teacher: base[idx]?.teacher ?? ''`
  → **정책 확정**: 기본 교사 필드는 **유지한다**. override는 과목만 바꾸는 케이스가 대부분이고, 동일 교사가 다른 과목을 대체하는 경우도 흔함. 테스트에서도 이 동작을 기대값으로 고정한다.

### 3.3 `filterOverridesInRange`

```ts
/** 날짜 범위 필터 (inclusive, YYYY-MM-DD 문자열 비교) */
export function filterOverridesInRange(
  overrides: readonly TimetableOverride[],
  from: string,
  to: string,
): readonly TimetableOverride[];
```

### 3.4 `dedupeOverridesKeepLatest`

```ts
/**
 * 같은 date+period 중복이 있을 때 가장 최신 하나만 남긴다.
 * 마이그레이션 1회용: 로드 시 적용하여 P3 (중복 저장 버그) 잔재를 정리한다.
 *
 * 승자 결정 규칙 (결정적, 타이브레이크 포함):
 *   1) max(updatedAt ?? createdAt) 최신 것이 승자
 *   2) 동률이면 id 사전순 가장 큰 것이 승자 (id는 `ovr-{timestamp}-{random}` 형식이라 사실상 유니크)
 *   3) 그래도 동률이면 원본 배열 뒤쪽 항목이 승자 (last-in-wins)
 */
export function dedupeOverridesKeepLatest(
  overrides: readonly TimetableOverride[],
): readonly TimetableOverride[];
```

### 4.2 `addOverride` 로직 변경

기존: `newOverrides = [...overrides, override]` → append-only (버그)

변경:
```ts
addOverride: async (input) => {
  const { overrides: current } = get();
  const { overrides: next, replacedId } = upsertOverride(
    current,
    input,
    new Date().toISOString(),
    () => `ovr-${Date.now()}-${Math.random().toString(36).slice(2, 8)}`,
  );
  set({ overrides: next });
  await scheduleRepository.saveTimetableOverrides({ overrides: next });
  return { replacedId };
},
```

호출부는 `replacedId`를 보고 Toast 문구를 다르게 표시 (선택적).

### 4.3 `updateOverride` 신규

```ts
updateOverride: async (id, patch) => {
  const { overrides: current } = get();
  const idx = current.findIndex((o) => o.id === id);
  if (idx < 0) return;

  // 도메인 경계 방어: date/period는 patch로 바꿀 수 없다.
  // 변경하려면 delete 후 addOverride(upsert)를 사용해야 한다.
  const { date: _d, period: _p, id: _id, createdAt: _c, ...safePatch } = patch as Record<string, unknown>;
  void _d; void _p; void _id; void _c;

  const updated: TimetableOverride = {
    ...current[idx]!,
    ...(safePatch as Partial<TimetableOverride>),
    id,                                             // 핵심 키 보호
    date: current[idx]!.date,                       // 핵심 키 보호
    period: current[idx]!.period,                   // 핵심 키 보호
    createdAt: current[idx]!.createdAt,             // 생성 시점 보호
    updatedAt: new Date().toISOString(),
  };
  const next = current.map((o, i) => (i === idx ? updated : o));
  set({ overrides: next });
  await scheduleRepository.saveTimetableOverrides({ overrides: next });
},
```

**정책 확정 (ADR)**:
- `updateOverride(id, patch)`는 `subject`, `classroom`, `reason`만 변경 가능하다.
- `date`/`period`는 **핵심 키**로 간주하여 스토어 경계에서 **strip**한다 (무결성 방어, UI readonly와 이중 방어).
- 날짜/교시를 바꾸려면 `deleteOverride(id)` → `addOverride({...new date/period})`를 순차 호출한다 (upsert 경로로 들어가므로 자동 무결성 보장).
- 타입 시그니처도 이에 맞춰 `Partial<Omit<TimetableOverride, 'id' | 'date' | 'period' | 'createdAt'>>`로 축소할 것.

### 4.4 `getEffectiveClassSchedule` 신규

```ts
getEffectiveClassSchedule: (date, weekendDays) => {
  const d = new Date(date + 'T00:00:00');
  const dayOfWeekVal = getDayOfWeek(d, weekendDays);
  if (!dayOfWeekVal) return [];
  const base = get().classSchedule[dayOfWeekVal] ?? [];
  const dayOverrides = get().overrides.filter((o) => o.date === date);
  return mergeOverridesIntoClassSchedule(base, dayOverrides);
},
```

### 4.5 로드 시 중복 정리

`load()` 안에서 `overridesData?.overrides` 로드 직후:

```ts
const loaded = overridesData?.overrides ?? [];
const deduped = dedupeOverridesKeepLatest(loaded);
if (deduped.length !== loaded.length) {
  // 마이그레이션 1회: 정리 결과를 저장소에 다시 쓴다
  await scheduleRepository.saveTimetableOverrides({ overrides: deduped });
}
set({ overrides: deduped, ... });
```

### 5.2 `TempChangeModal` 수정

**Props 확정 — Discriminated Union (Warning 1 대응)**:

```ts
// 공통 필드
interface TempChangeModalBaseProps {
  date: string;
  period: number;
  currentSubject: string;
  currentClassroom?: string;
  onClose: () => void;
}

// create 모드 — onSave 필수
interface TempChangeModalCreateProps extends TempChangeModalBaseProps {
  mode?: 'create';                                 // 기본값
  onSave: (override: Omit<TimetableOverride, 'id' | 'createdAt' | 'updatedAt'>) => void;
}

// edit 모드 — initialOverride + onUpdate 필수, onSave는 존재하지 않음
interface TempChangeModalEditProps extends TempChangeModalBaseProps {
  mode: 'edit';
  initialOverride: TimetableOverride;
  onUpdate: (id: string, patch: Partial<Omit<TimetableOverride, 'id' | 'date' | 'period' | 'createdAt'>>) => void;
}

export type TempChangeModalProps = TempChangeModalCreateProps | TempChangeModalEditProps;
```

**컴파일 타임 보장**: TypeScript 타입 판별로 edit 렌더 시 `onSave={() => {}}` 더미를 넘기지 않아도 됨. create 사이트에서는 `onUpdate` 미제공 허용.

**내부 동작**:
- `props.mode === 'edit'`:
  - 제목: "임시 시간표 수정"
  - date/period는 **텍스트 표시**만 (input 아님)
  - subject/classroom/reason 초기값은 `props.initialOverride`에서 읽음
  - 저장 버튼 → `props.onUpdate(props.initialOverride.id, { subject, classroom, reason })`
- 그 외 (`'create'` 또는 undefined):
  - 기존 동작 유지 (`props.onSave(...)`)

### 5.4 기존 우클릭 진입점

- `create` 모달을 바로 여는 현 동작 유지
- 단, override가 이미 있는 셀 우클릭 시 현재는 즉시 삭제 → **컨텍스트 메뉴로 변경** (수정 / 삭제 / 이 날의 모든 변동 보기)
- 컨텍스트 메뉴는 MVP 범위 밖. Phase 1에서는 기존 동작 유지하되, override가 있는 셀을 우클릭하면 **삭제 대신 edit 모달 열기**로 변경 (사용자가 실수로 날리는 현상 방지)

### 6.1 신규 변동 추가 (패널 경유)
1. 시간표 페이지 → `[변동 시간표]` 클릭 → 드로어 열림
2. `[+ 변동 추가]` 클릭 → 빈 `TempChangeModal` (오늘 날짜, 1교시 기본)
3. 저장 → 스토어 upsert → 드로어 리스트에 즉시 반영

### 6.2 셀 우클릭 추가 (기존 동작)
1. 시간표 셀 우클릭 → 해당 날짜/교시 자동 입력된 `TempChangeModal`
2. 저장 → 동일

### 6.3 수정
1. 드로어 리스트에서 `[수정]` 클릭 또는 override 있는 셀 우클릭
2. `TempChangeModal` (mode=edit, date/period readonly)
3. 저장 → `updateOverride`

### 6.4 삭제
1. 드로어 리스트에서 `[삭제]` 클릭
2. 확인 다이얼로그 (선택)
3. `deleteOverride`

### 6.5 NEIS 재동기화
1. 기존 NEIS 불러오기 버튼 클릭 → `saveClassSchedule`/`saveTeacherSchedule`만 호출
2. overrides는 별도 키라 영향 없음
3. 토스트: "기본 시간표가 갱신되었습니다. 등록된 변동 시간표 N건은 유지됩니다." (신규)

## 8. Edge Cases & Policies

| 케이스 | 정책 |
|---|---|
| 과거 날짜에 override 추가 | 허용 (기록 목적) |
| 오늘 미래 override 여러 개 | 허용, 뱃지 카운트에 포함 |
| `subject=""` + `classroom` 입력 | 공강 간주, classroom은 무시 (또는 "자습 @교실" 표시) → 후자 채택 |
| `subject=""` + `reason=""` | 사유 없는 자습 허용 |
| override date가 주말인데 주말이 비활성 요일 | `getEffectiveXSchedule` 빈 배열 반환 → 리스트는 표시되지만 시간표 뷰는 영향 없음. MVP에서는 허용 |
| period가 `classSchedule[day].length`보다 큼 | 무시. 리스트에는 표시하되 "교시 범위 초과" 툴팁 (선택) |
| 기간 단위 override (Phase 3) | MVP 범위 밖. 단, 데이터 모델에 영향 없도록 현 스키마 유지 |

## 9. Risks & Mitigations

| 리스크 | 확률 | 영향 | 완화 |
|---|---|---|---|
| 기존 사용자 JSON에 중복 override 다수 잔존 | 중 | 중 | `load()` 시 `dedupeOverridesKeepLatest` 자동 적용 + 저장소 재기록. 테스트 S7로 검증 |
| `updateOverride`에서 date/period를 바꾸는 patch → upsert 충돌 | 저 | 중 | UI에서 edit 모드 date/period readonly. 코드에서는 patch 허용하되 warn 로그 |
| `getEffectiveClassSchedule` 도입으로 기존 뷰 회귀 | 중 | 중 | 우선 class 탭만 전환, 순차 확장. 단위 테스트 + 수동 S3, S4 |
| NEIS 자동 동기화가 overrides를 건드림 | 저 | 고 | `saveClassSchedule/saveTeacherSchedule`가 `timetable-overrides` 키를 건드리지 않음을 코드 리뷰로 재확인 + 명시 테스트 S5 |
| Drive 충돌 (두 기기 동시 override 편집) | 저 | 저 | 기존 `SyncToCloud` 마지막-쓰기-승 동작 그대로. MVP 범위 밖 |
| 사용자가 뱃지 카운트를 "알림"으로 오해 | 저 | 저 | 툴팁 "미래 변동 N건" 명시 |

## 11. Implementation Order (Phase 1)

1. domain: `TimetableOverride.updatedAt?` 추가
2. domain/rules: 4개 순수 함수 + 단위 테스트 (먼저 빨갛게, 구현 후 초록)
3. store: `addOverride` upsert 전환, `updateOverride`/`getOverridesInRange`/`getEffectiveClassSchedule` 신설
4. store: `load()` dedup 마이그레이션
5. UI: `TempChangeModal` edit 모드 props
6. UI: `TimetableOverridesPanel` 신규
7. UI: `TimetablePage` 툴바 버튼 + 패널 상태
8. 수동 S1~S5 검증
9. Phase 2: `DashboardTimetable` class 탭 + `TodayProgress` + 기타 뷰에서 `getEffectiveClassSchedule`로 전환
10. 수동 S3~S9 최종 검증
