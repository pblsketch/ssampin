## 0. 한 줄 요약

데이터 모델·저장소·동기화·모달 UI가 **이미 구현되어 있으나** 발견성이 낮고 적용 범위가 불균일한 `TimetableOverride` 기능을 **정식 기능으로 승격**한다. 신규 개발이 아니라 **노출·일원화·무결성 보완**이 핵심.

## 5. 데이터 모델 변경 (Phase 1)

### 5-1. 엔티티 변경 (선택적)
현 `TimetableOverride`는 충분하나, 수정 이력 추적이 필요하면 `updatedAt` 추가.

```ts
// 안 1 (최소 변경): 그대로 사용. 수정은 delete+add로 처리
// 안 2 (권장): updatedAt 추가 + updateOverride API 신설
export interface TimetableOverride {
  readonly id: string;
  readonly date: string;
  readonly period: number;
  readonly subject: string;
  readonly classroom?: string;
  readonly reason?: string;
  readonly createdAt: string;
  readonly updatedAt?: string;   // 신규 (optional, 기존 데이터 호환)
}
```
→ **안 2 채택**. 저장 포맷 호환(optional 필드), 수정 시점 추적 가능.

### 5-2. 스토어 API 변경

| API | Phase | 변경 내용 |
|---|---|---|
| `addOverride(input)` | 1 | **upsert**: `date+period` 일치 시 기존 교체 |
| `updateOverride(id, patch)` | 1 | 신설 (권장) — 부분 갱신 + `updatedAt` 기록 |
| `deleteOverride(id)` | — | 변경 없음 |
| `getOverridesForDate(date)` | — | 변경 없음 |
| `getOverridesInRange(from, to)` | 1 | 신설 — 관리 패널용 |
| `getEffectiveTeacherSchedule(date)` | — | 변경 없음 |
| `getEffectiveClassSchedule(date)` | 2 | 신설 |

## 7. 충돌/우선순위 규칙 (P5 대응)

### 7-1. NEIS 자동 동기화 시나리오
NEIS에서 학교 시간표를 다시 불러올 때:

| 소스 | 대상 | 규칙 |
|---|---|---|
| NEIS 기본 시간표 | `classSchedule`, `teacherSchedule` | **덮어씀** (원래 동작) |
| 사용자 override | `overrides` | **보존** — 절대 자동 삭제 금지 |

### 7-2. 유효 시간표 계산 우선순위
```
표시값 = override(date, period) 존재? → override 값
                                      → 아니면 base schedule 값
```
단순 규칙. base가 바뀌어도 override는 살아남음.

### 7-3. 사용자 안내
- NEIS 재동기화 버튼 근처에 "등록된 변동 시간표 N건은 유지됩니다" 툴팁
- 과거 override 자동 정리는 **안 함** (사용자가 기록 보존 의도일 수 있음). 단 Phase 3에서 "30일 이전 정리" 옵션 검토

## 9. 테스트 전략

### 9-1. 단위 테스트 (vitest)
- `timetableRules.mergeOverrides()`
  - 빈 overrides → base 그대로
  - 같은 date+period 1건 → 치환
  - 공강(subject="") → null
  - period 인덱스 경계 (0, max, 음수)
- `useScheduleStore.addOverride` upsert
  - 같은 date+period 재호출 시 배열 길이 불변
  - `updatedAt` 갱신 확인
- `useScheduleStore.getEffectiveClassSchedule`
  - 교사 버전과 동일 입력에 동일 패턴 결과

### 9-2. 통합 시나리오 (수동)
1. 시간표 페이지 → 변동 추가 → 목록에 나타남 → 대시보드 반영 확인
2. 같은 셀 다시 추가 → 항목 개수 동일, 값만 변경
3. NEIS 재동기화 → override 건재 확인
4. Google Drive 동기화 → 다른 기기에서 override 확인
5. 삭제 → 모든 관련 화면에서 원복

### 9-3. 런타임 검증 원칙 (feedback_runtime_verification 준수)
"동작한다" = 토스트 표시가 아니라 **실제 JSON 파일**에 저장된 내용 확인 + **재시작 후**에도 유지.
