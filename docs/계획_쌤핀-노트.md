## 1. 개요 & 목표 & 포스트잇 메모와의 구분

### 1-1. 개요

쌤핀 앱에 **노션 스타일 블록 기반 노트 에디터**를 새 메뉴로 신설한다. 이름은 **"쌤핀 노트"**.
기존 포스트잇 메모와는 **완전히 별개**의 기능이며, 사이드바에서 각각의 메뉴로 공존한다.

### 1-2. 왜 포스트잇만으로 부족한가

| 사용자 니즈 | 포스트잇 메모 | 쌤핀 노트 |
|---|---|---|
| "내일 수업 준비를 메모" | OK (1~3줄) | 과함 |
| "한 학기 생활지도 방침을 체계적으로 기록" | 부족 — 길이·구조 한계 | OK |
| "학급운영 레시피/수업 자료를 섹션별로 정리" | 불가 | OK |
| "`@김철수` 학생 멘션으로 바로 프로필 연결" | 불가 | OK (차별화) |
| "제목·목차·체크리스트·표·인용 혼합 장문 문서" | 불가 | OK |

### 1-3. 핵심 목표

1. **장문 기록 지원**: 수업일지, 학급경영 노하우, 진로상담 스크립트 같은 체계적 문서 작성.
2. **위노트 경쟁 우위 확보**: 위노트의 노트 기능(섹션→페이지→블록) 전부 커버 + 쌤핀만의 차별화 블록(학생 멘션, 담임메모 임베드, 학급 컨텍스트).
3. **오프라인 완전 동작**: 기존 `IStoragePort`를 재사용해 Electron/브라우저 모두에서 동일 저장소 추상화 유지.
4. **Clean Architecture 준수**: domain ← usecases ← adapters ← infrastructure 의존성 방향 엄수.
5. **한국어 IME 완전 대응**: 조합 중인 한글(한/영 IME)이 올바르게 동작해야 함.

### 1-4. 포스트잇 메모와 쌤핀 노트의 구분 (사용자 관점)

| 차원 | 포스트잇 메모 (기존) | 쌤핀 노트 (신규) |
|---|---|---|
| 은유 | "책상 위 포스트잇" | "바인더/공책" |
| 수명 | 짧음 (휘발성 아이디어) | 김 (누적 기록) |
| 구조 | 평면 (자유 배치) | 계층 (노트북→섹션→페이지→블록) |
| 내용량 | 1~10줄 | 무제한 |
| 편집 UI | 카드 표면 인라인 | 전용 4단 레이아웃 |
| 태그/검색 | 없음 | 있음 (태그 + 전문검색) |
| 학생 연결 | 없음 | 있음 (`@학생` 멘션) |
| 데이터 포맷 | 단일 마크다운 문자열 | 블록 트리 (JSON) |

**사이드바 배치**: 메모 항목 바로 아래에 "쌤핀 노트" 추가.

## 2. 사용자 시나리오

### 시나리오 A. 1학기 학급경영 플래너

> 중학교 1학년 담임 김선생은 새 학기를 앞두고 **"2026학년도 1학기 학급경영"** 이라는 노트북을 만든다.
> 섹션으로 **①학급 규칙 ②월별 계획 ③학부모 소통 ④생활지도 원칙**을 나눈다.
> "학급 규칙" 섹션 안에 "입학 첫날 오리엔테이션" 페이지를 만들고
> H1 제목, 체크리스트(준비물), Callout(중요 공지), 표(시간표)를 섞어 문서를 완성한다.
> 필요시 `/기록삽입`으로 작년 동일 이벤트 담임메모를 불러와 참고한다.

### 시나리오 B. 진로상담 스크립트 라이브러리

> 진로교사 박선생은 **"진로상담 스크립트"** 노트북에 학년별로 섹션을 만들고
> 각 섹션에 "IT 관심 학생", "체육계열 지망", "진로 미정" 페이지를 준비한다.
> 상담 진행 중 `@김철수`를 타이핑해 해당 학생 프로필로 점프해서 기존 기록을 확인하고
> 상담 후에는 Callout(결론)에 다음 액션을 기록한 뒤 Markdown으로 내보내 외부에 제출한다.

### 시나리오 C. 주간 수업일지

> 국어과 이선생은 **"수업일지 2026"** 노트북 → 주차 섹션에 매주 페이지를 추가.
> 페이지 제목 = `2026-W17 (4/22~4/26)`, 블록으로 H2(차시별 요약) + 체크리스트(과제) + 인용(학생 명언) + 이미지(판서 사진)를 삽입.
> 태그 `#국어 #1학년 #문법`으로 분류해 주말에 전문검색으로 한 학기 일지를 한눈에 훑는다.

### 시나리오 D. 수업 자료 아카이브

> 수학교사 최선생은 **"문제 은행"** 노트북 단원별 섹션 안에 자주 내는 문제를 블록으로 정리.
> 코드 블록에 TeX 수식 대신 텍스트 수식을 보관, 표로 정답/해설을 옆에 나란히 둠.
> 다음 학기 같은 단원 수업 때 페이지를 복제해 재활용.

### 시나리오 E. 학생 관찰 누적 기록 (차별화 핵심)

> 담임 조선생은 특정 학생에 대한 관찰을 **쌤핀 노트**에 누적 기록하되
> 민감 정보는 담임메모(기존 StudentRecord)에 남기고, 노트에서 `/기록삽입`으로 참조만 임베드한다.
> "노트 = 공유 가능한 정리 문서" / "담임메모 = 비공개 민감 기록" 의 역할 분담.

### 3-2. 한국어 IME 관점 주의 (매우 중요)

- **ProseMirror 계열 (TipTap / BlockNote / Remirror)**: 2024~2025년 Chrome 업데이트 이후 한국어 IME 이슈가 **보고된 적 있음** ([ProseMirror Issue #1484](https://github.com/ProseMirror/prosemirror/issues/1484)). 현재는 대체로 안정. QA 필수.
- **Lexical**: Meta가 한국어 IME 버그를 **적극적으로 수정** 중 ("Prevent typeahead menu from closing during IME composition", "Fix selected text not properly deleted after IME input on Safari"). Meta 내부에서 한국어 사용자 대응이 되고 있는 것으로 추정.
- **Slate.js**: contentEditable을 얇게 래핑해 과거부터 한글 IME 이슈가 잦았음. Plate도 Slate 기반이라 동일 리스크 상속.
- **Editor.js**: React 통합이 비공식(`react-editor-js`). IME 상황 확인 필요.

### 3-3. 주요 비교축 상세

#### 슬래시(`/`) 커맨드
- **기본 내장**: BlockNote, Plate, Editor.js, novel.sh, Yoopta
- **확장으로 구현**: TipTap (공식 예제 + `@harshtalks/slash-tiptap` 커뮤니티 패키지 가용), Lexical, Remirror, Slate

#### 드래그 핸들(⠿)
- **기본 내장**: BlockNote, Yoopta, Plate(shadcn 블록)
- **확장/구현 필요**: TipTap, Lexical, Slate, Editor.js

#### JSON 직렬화
- 모두 지원. BlockNote는 `editor.document`가 **lossless JSON** 권장, Markdown 변환은 **lossy**.

#### 커스텀 블록 확장 성숙도
- 최상급: **TipTap** (ProseMirror 노드 정의 그대로 노출)
- 상급: BlockNote (TipTap 확장 + `schema.extend`로 커스텀 블록)
- 중급: Lexical (Node 클래스 상속), Plate, Yoopta
- 하급: Editor.js (블록 단위 플러그인, 복잡한 인라인 확장은 제약)

#### Markdown 지원
- BlockNote: `blocksToMarkdownLossy` / `tryParseMarkdownToBlocks` **공식 제공** (lossy)
- TipTap: `@tiptap/extension-markdown` 또는 `prosemirror-markdown` 활용
- Editor.js: 플러그인 필요
- 그 외 대부분: 직접 구현 필요

#### 라이선스 주의
- **BlockNote = MPL-2.0**: 상업 사용 OK, 수정 시 동일 라이선스 공개 의무 (쌤핀은 수정하지 않고 API로 소비하므로 문제없음).
- **Plate Pro 유료 모듈 주의**: 필요한 기능이 Pro에 있는지 확인.

## 4. 권장 스택 & 근거

### 4-1. 1차 권장: **BlockNote** (TipTap 위에 얹힌 Notion 스타일 기본 내장)

**추천 근거**:
1. **개발 속도**: Notion 스타일 UI (슬래시 메뉴 / 드래그 핸들 / 플로팅 툴바)가 **기본 내장** → Phase 1 MVP 2주 목표 달성 현실적.
2. **커스텀 블록 확장성**: 내부가 TipTap/ProseMirror이므로 쌤핀 차별화 블록(학생 멘션, 담임메모 임베드)을 TipTap 확장 방식으로 구현 가능.
3. **JSON 기반 lossless 저장**: `editor.document`가 저장 포맷 그대로 — `notes/{pageId}.json` 저장 전략과 궁합 좋음.
4. **활발한 유지보수**: 2026-04 기준 v0.47.3 릴리즈 (16일 전). 월간 릴리즈 페이스.
5. **라이선스 적합**: MPL-2.0 — 쌤핀 본체(GPL-3.0)에서 **수정 없이 API 소비만** 하면 제약 없음 (MPL은 file-level copyleft이며 Category B 호환). 법률 메모: BlockNote 소스를 포크/수정해 배포할 경우에만 MPL 의무 발생.

**버전 pin 전략 (v0.2 확정)**:
```jsonc
// package.json dependencies
"@blocknote/core": "~0.47.3",     // patch만 자동 업
"@blocknote/react": "~0.47.3",
"@blocknote/mantine": "~0.47.3"   // or "@blocknote/shadcn"
```
- `~` range = **patch만** 자동 업그레이드 (0.47.3 → 0.47.99 OK, 0.48.x 차단)
- 이유: v0.x는 minor 릴리즈마다 파괴적 변경 가능 (R3). minor 업그레이드는 별도 스프린트로.

**리스크**:
- 번들 사이즈 (Mantine UI 포함 시 ~250KB). Electron 앱이라 허용 범위 (인스톨러 ~200MB 중 미미).
- 한국어 IME: BlockNote 고유 이슈는 명시 보고 없음. TipTap/ProseMirror 이슈가 상속될 수 있어 **MVP 초반 IME QA 필수**.
- `shadcn/ui` 스타일 지원도 있으나, 쌤핀은 Tailwind를 쓰므로 headless 모드 + 자체 스타일링 검토.

## 5. 데이터 모델 (TypeScript 인터페이스)

### 5-0. 에디터-agnostic 경계 원칙 (v0.2 추가)

R1(BlockNote IME 치명 이슈 시 TipTap으로 후퇴) 완화책을 **구조적으로 강제**하기 위한 원칙:

1. **`domain/` 레이어는 BlockNote/TipTap 타입을 직접 import 금지**
   - `NoteBlock`, `NoteInlineContent` 등 도메인 타입은 **쌤핀 자체 스키마**만 표현.
   - BlockNote의 `Block` / `PartialBlock` / `DefaultBlockSchema` 등은 `adapters/` 이하에서만 사용.
2. **BlockNote 특화 필드는 `props: Record<string, unknown>` 에 옮겨담기**
   - 예: 표 cell merge 정보, BlockNote 내부 id 등은 도메인이 해석하지 않고 불투명 보존.
3. **변환은 `adapters/presenters/notePresenter.ts` 단일 창구**
   - `toBlockNoteDocument(note: NoteDocument)` / `fromBlockNoteDocument(doc: BlockNoteDocument)` 두 함수만 BlockNote 타입에 의존.
   - 교체 시 이 파일만 다시 쓰면 됨.
4. **`usecases/` 는 `NoteDocument` 를 불투명(opaque)하게 다룸**
   - UpdatePageContent, DuplicatePage 등은 내용 해석 없이 통째로 read/write만 수행.

→ 이 원칙을 지키면 BlockNote → TipTap 교체 시 `adapters/presenters` + `adapters/components/Note/Editor/*` 만 고쳐도 됨. `domain/`, `usecases/`, Repository 구현 전체는 영향 없음.

export function sortNotebooksForSidebar(list: readonly Notebook[]): readonly Notebook[] {

### 7-3. 블록 호버 UI (Notion 패턴)

```
⠿  + │ 이 블록이 본문입니다.
     │
⠿  + │ 다른 블록
```

- 호버 시 좌측에 `⠿` (드래그 핸들) + `+` (새 블록 추가 메뉴) 노출.
- `+` 클릭 = 슬래시 메뉴 자동 오픈.
- BlockNote는 이 UI를 **기본 제공**하므로 스타일만 쌤핀 톤으로 오버라이드.

### 7-4. 자동 저장 배지

에디터 우상단에 항상 표시:
- `저장 중...` (saving) — `sp-muted` + 작은 스피너
- `저장됨 · 15:42` (saved) — `text-emerald-400` + 체크 아이콘
- `저장 실패 · 재시도` (error) — `text-red-400` + 재시도 버튼

### 7-5. 폰트 규칙

- 제목: Noto Sans KR Bold, 24px/32px
- H1/H2/H3: Bold, 28/22/18px
- 본문: Regular, 15px, line-height 1.7
- 코드 블록: `JetBrains Mono` (번들) fallback → `monospace`, 14px

## 9. 쌤핀 차별화 블록 상세

### 9-1. 학생 멘션 `@학생이름` (★ 핵심 차별화)

**동작**:
1. 사용자가 `@` 키 입력 → 드롭다운 팝업.
2. 현재 노트북의 `context.teachingClassId`로 연결된 학급의 재학(`status: active`) 학생 목록 표시.
3. 타이핑하면 한글 초성 검색(`ㄱㅊㅅ` → `김철수`) 지원 (Phase 3 이후).
4. 선택 시 인라인 멘션 노드 삽입: `mentionId=<studentId>`, 렌더는 `@김철수` 파란 칩.
5. 클릭 시 학생 상세(기존 `/student-records` 경로)로 이동.

**구현 메모**:
- BlockNote는 **inline content extension**을 지원 — TipTap의 `@tiptap/extension-mention`을 그대로 얹을 수 있음.
- 멘션 드롭다운은 `tippy.js`로 위치 계산 (TipTap/BlockNote 내장).
- 학생이 학급에서 빠진 경우 멘션은 유지되지만 색상 회색 + 툴팁 "이 학생은 더 이상 학급에 없습니다".

**데이터 모델**:
```ts
// NoteInlineContent의 mention 타입
{
  type: 'mention',
  mentionType: 'student',
  mentionId: 'stud-abc',
  // 렌더 시 학생 이름은 실시간 조회 (이름 변경 대비)
}
```

### 9-2. 담임메모 임베드 `/기록삽입`

**동작**:
1. `/기록삽입` 선택 → 모달로 학생 + 날짜 범위 필터.
2. 선택된 StudentRecord들을 읽기 전용 카드 블록으로 임베드.
3. 원본 담임메모가 변경되면 임베드 블록도 자동 갱신 (reference by ID).
4. 삭제된 원본은 "원본이 삭제되었습니다" placeholder.

**속성**:
```ts
{
  type: 'recordEmbed',
  props: {
    recordIds: ['rec-1', 'rec-2'],
    displayMode: 'summary' | 'full',
    hidePrivate: true,  // 출력 시 비공개 기록 제외
  }
}
```

**중요**: 담임메모는 민감 정보이므로 **노트 Markdown 내보내기 시** recordEmbed 블록은 기본적으로 제외(또는 "[담임메모 {n}건]" 으로 익명화) — 사용자 옵션으로 제어.

### 9-3. Callout 박스

5가지 종류 (`CalloutKind`): info / warning / tip / note / danger.

**스타일**:
```
┌─────────────────────────────┐
│ ℹ️  정보                     │
│ 본문 내용...                 │
└─────────────────────────────┘
```

- 왼쪽 4px 컬러 바 + 배경 틴트 + 이모지 아이콘.
- 색상은 디자인 토큰: info=sp-accent, warning=sp-highlight, danger=red-500, tip=emerald-400, note=sp-muted.

**구현**: BlockNote의 `createBlockSpec` API로 커스텀 블록 등록.

### 9-4. 학급 컨텍스트 인식

노트북 생성 시 `context.teachingClassId`를 선택하면 해당 노트북의 모든 페이지에서:
- `@` 자동완성이 해당 학급 학생만 표시.
- `/기록삽입` 시 해당 학급 기본 필터.
- Phase 4+ 에서 "이 학급에 새 학생이 추가됨" 같은 알림도 가능.

## 10. 검색 & 태그 구현

### 10-1. 검색 전략 선정

| 선택지 | 장점 | 단점 | 쌤핀 적합도 |
|---|---|---|---|
| **FlexSearch** | CJK(한중일) 토크나이저 내장, 압도적 속도, 의존성 0 | 러닝 커브 | ★★★★★ |
| Lunr.js | 단순, 공식 문서 풍부 | 한국어 기본 지원 약함 (커스텀 토크나이저 필요) | ★★★ |
| Fuse.js | 퍼지 매칭 간편 | 전문검색 성능 낮음, 블록 단위 정밀도 부족 | ★★ |
| 자체 구현 | 완전 제어 | 공수 +2주 | ★★ |

→ **FlexSearch 채택**. `Charset.CJK` 옵션으로 한국어 즉시 지원.

### 10-2. 인덱스 구조

블록 단위가 아니라 **페이지 단위** 인덱싱 (블록 단위는 Phase 5+):

```ts
const index = new FlexSearch.Document({
  tokenize: 'forward',
  charset: 'cjk',
  document: {
    id: 'id',
    index: ['title', 'plainText', 'tags'],
    store: ['title', 'sectionId', 'notebookId', 'snippet'],
  },
});
```

- `plainText`: 블록 트리를 평문으로 직렬화 (`noteBlockRules.ts#extractPlainText`).
- 페이지 저장 시 인덱스 갱신, 앱 로드 시 전체 인덱스 리빌드 (페이지 1000개까지 수 초 내).
- 인덱스는 **휘발성** — `indexedDB`에 캐시도 가능하나 MVP는 메모리만.

### 10-3. 검색 UX

- 노트북 컬럼 상단 검색창 (`Ctrl+Shift+F`로도 오픈).
- 결과는 스니펫(검색어 주변 40자 ± 하이라이트) + 경로("학급경영 > 월별계획 > 3월")로 표시.
- 결과 클릭 시 해당 페이지로 이동 + 검색어 하이라이트.

### 10-4. 태그

- 페이지당 태그 배열 (`NotePage.tags`).
- 편집 UI: 페이지 제목 아래 `+태그` 버튼 → 드롭다운.
- **자동완성**: 기존 태그 전체 집합을 추천.
- 태그 클릭 → 해당 태그로 필터된 페이지 목록 (별도 뷰).
- **과목별 컬러**: CLAUDE.md의 과목 컬러 매핑 재사용 (`#국어` → yellow 칩).

## 11. 이미지/첨부파일 처리

### 11-1. 저장 위치 전략

| 환경 | 경로 | 참조 방식 |
|---|---|---|
| Electron | `app.getPath('userData')/note-assets/{pageId}/{fileId}.{ext}` | `file://...` 또는 custom protocol |
| 브라우저(개발) | `localStorage` 용량 한계 → **base64 inline** | `data:image/...` |

### 11-2. Electron에서의 권장: Custom Protocol

`file://` 프로토콜을 직접 쓰면 CSP 이슈가 있음. Electron 메인에서 `protocol.handle('ssampin-asset', ...)`을 등록해:

```
ssampin-asset://note/{pageId}/{fileId}.png
```

를 실제 userData 파일로 해석. 렌더러에서는 일반 URL처럼 사용.

**v0.2 결정 — MVP에서도 분리 저장 (base64 인라인 금지)**

**이유**:
- 쌤핀 노트는 포스트잇과 달리 **장문 문서**로 페이지 JSON이 커짐. base64 이미지 여러 장을 같은 JSON에 포함하면 페이지 여는 순간 매번 전체를 파싱해야 함 (수 MB 페이지 시 체감 지연).
- 쓸 때마다 전체 본문이 다시 쓰이는 autosave와 맞물리면 **이미지 포함 페이지는 타이핑마다 수 MB write** → 디스크 I/O 폭발.
- 포스트잇(`MemoImage`)은 단일 카드 단위라 base64가 합리적이지만, 노트는 확장성 프로파일이 전혀 다름.

**MVP 구현 전략 (Phase 1 범위)**:

| 환경 | 저장 경로 | 참조 방식 |
|---|---|---|
| Electron | `userData/data/note-assets/{pageId}/{fileId}.{ext}` (파일 분리) | `ssampin-asset://note/{pageId}/{fileId}.png` custom protocol |
| 브라우저(개발) | `IndexedDB` (이미 `idb` 의존성 있음, [`package.json:45`](../package.json#L45)) | `blob:` URL (런타임 생성) |

- Electron custom protocol 등록은 `electron/main.ts` 에 ~30 LOC. Phase 1에 포함시켜도 부담 작음.
- 브라우저 폴백은 `idb` 활용 (localStorage 용량 한계 회피).
- 이미지 블록의 도메인 표현은 **참조만 저장**:
  ```ts
  { type: 'image', props: { assetRef: 'ssampin-asset://note/pg-1/img-1.png', width: 800, height: 600, alt: '' } }
  ```
- 페이지 삭제 시 `note-assets/{pageId}/` 디렉토리 전체 cleanup.

**포스트잇과의 차이 요약**: 포스트잇은 base64 유지 (단일 카드라 부담 적음), 쌤핀 노트는 **파일 분리** (장문 확장성).

### 11-3. 파일 크기 제한

- 이미지: 5MB (MemoImage 정책과 동일) + 자동 리사이즈 (긴 변 2048px).
- 원본 보존 옵션은 Phase 5+ 검토.

### 11-4. 붙여넣기/드래그

- 이미지 클립보드 붙여넣기 → 자동으로 이미지 블록 삽입 (BlockNote 기본 지원).
- 파일 드래그 앤 드롭 → 에디터 영역 위에서 hover 시 drop zone 오버레이.

### 14-2. 필수 테스트 케이스

1. **한국어 IME 시나리오** — 자동화 어려움. 수동 QA 체크리스트 작성:
   - "가나다 입력 → 한/영 토글 → abc 입력" 반복
   - 조합 중 슬래시 메뉴 오픈/닫기 타이밍
   - `@` 입력 후 한글 학생 이름 타이핑
   - macOS / Windows 양쪽에서 Chromium 기본 IME + Windows MS IME + macOS 2벌식
2. **자동 저장 race condition**: 1초 debounce 중 연속 수정 → 최종 상태만 저장되는지.
3. **대용량 페이지**: 10,000자 본문 자동 저장 지연 측정 (목표: <500ms).
4. **파일 I/O 실패**: 저장 실패 시 "저장 실패 · 재시도" 배지 노출 + 다음 변경 시 자동 재시도.
5. **학생 멘션 생명주기**:
   - 멘션된 학생이 학급에서 빠짐 → 멘션 회색 + 툴팁.
   - 학생 이름 변경 → 멘션 표시 즉시 업데이트.
   - 학생 삭제(hard delete) → 멘션 유지 + "삭제된 학생" 표시.
6. **블록 드래그**: 블록 이동 후 자동 저장 → 재로드 시 순서 보존.
7. **Markdown 라운드트립**: export → import (손실 허용 범위) 문서화.
8. **블록 스키마 마이그레이션**: 향후 타입 추가 시 `noteBlockRules.ts#migrate`가 미지원 블록을 `paragraph`로 graceful degrade.

### 14-3. 한국어 IME 수동 QA 체크리스트 (템플릿)
